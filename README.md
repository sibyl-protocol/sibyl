# Sibyl — AI Oracle × Prediction Market

> 複数のAIエージェントが合議して「判断」を下すオラクル。最初のユースケースとして予測市場を自ら構築する。

## コンセプト

既存のオラクル（Chainlink等）は構造化データ（価格フィード等）しか扱えない。
UMAは人間の投票で判断するが、遅くてコストが高い。

Sibylは **非構造化データに対してAIが判断を下すオラクル**。
「事実」ではなく「解釈」を扱える。

### 名前の由来
- 古代ギリシャの巫女（Sibyl）— 神託を下す存在
- Psycho-Pass のシビュラシステム — 複数の脳が集合的に判断を下す
- 複数AIエージェントの合議 → コンセプトそのもの

## ポジショニング

| | Chainlink | UMA | Sibyl |
|---|---|---|---|
| データ型 | 構造化（価格等） | 人間の紛争解決 | 非構造化 + 判断 |
| 速度 | 速い | 遅い（数日〜週） | 数分 |
| コスト | 低 | 高（人間コスト） | 中間 |
| 判断力 | なし | 人間依存 | AI合議 |
| 得意領域 | 明確な事実 | 紛争解決 | 解釈が分かれるもの |

## Polymarketとの差別化

| | Polymarket | Sibyl |
|---|---|---|
| 判定 | UMA（人間投票、数日） | AI（数分）+ 人間バックストップ |
| 曖昧な解決 | 頻繁に揉める | AIが文脈理解して判定 |
| マーケット作成 | 運営管理 | 将来的には誰でも（AI判定可能なら） |
| 解決速度 | 数日〜数週間 | 数分〜24h |

**「速い」だけじゃなく「曖昧なイベントも解決できる」のが本質的な差別化。**

## アーキテクチャ

### フロー

```
ユーザーがマーケットにベット（Yes/No）
    ↓
期限到来
    ↓
AI Oracle 判定（複数エージェント合議、数分）
    ↓
異議申立期間（24h）
    ↓
異議なし → 自動精算
異議あり → 人間委員会（v2）
```

### Oracle 設計

```
┌─────────────────────────────────┐
│        Sibyl Oracle Network      │
├─────────────────────────────────┤
│  Agent A ──┐                    │
│  Agent B ──┼─→ Aggregation ─→ Chain
│  Agent C ──┘   (加重平均)       │
├─────────────────────────────────┤
│  重み: 実績ベース（decay付き）    │
│  多様性: 異なるモデル/手法        │
│  透明性: 各エージェントの判断を公開│
└─────────────────────────────────┘
```

### 設計方針

| 要素 | 方針 | 備考 |
|---|---|---|
| 参加モデル | Permissioned（運営管理） | Chainlinkも実態はsemi-permissioned |
| 合意方式 | 加重平均 + 実績ベース重み付け | Decay導入で古参支配を防止 |
| エージェント多様性 | インセンティブ設計で促す | 同質回答の報酬を薄く。将来的にTEEも検討 |
| 安全弁 | 楽観的オラクル式 | 通常はAI即決、異議時のみ人間エスカレーション |
| 信頼の源泉 | 判定精度の実績 + 透明性 | 分散性ではなく品質で勝負 |

## チェーン

**Solana**
- 速い・安い → 小額ベットに向いている
- Rust/Anchor でスマコン開発
- 予測市場の先行事例あり

## ターゲットマーケット（ジャンル）

解釈が分かれる領域を狙う:
- **クリプト/テック** — 「ロードマップ通りに実装されたか」「Breaking Changeに該当するか」
- **政治/規制** — 「この法案はDeFi規制に該当するか」
- **地政学** — 「この制裁は実質的影響があるか」

最初はクリプト/テック領域から。ユーザー層がクリプトネイティブと重なる。

## MVP スコープ

### 含む
- マーケット作成（Yes/No形式、運営が作成）
- ベット（SBYLトークン）
- AI判定（2-3エージェントの合議）
- 自動精算
- オリジナルトークン（SBYL）発行・利用

### 含まない（v2以降）
- ユーザーによる自由なマーケット作成
- 異議申立 + 人間エスカレーション
- 複雑な合議メカニズム

### 技術スタック（想定）

```
Frontend:  Next.js（シンプルなベットUI）
Contract:  Solana / Anchor（マーケット/ベット/精算）
Backend:   AI判定サービス（複数LLM + 合議ロジック）
Oracle:    判定結果をオンチェーンに書き込むbot
```

### 最初のマーケット
**「CLARITY Act (H.R.3633) は2026年8月31日までに大統領の署名を受けて成立するか？」**

判定基準:
- Yes: H.R.3633 またはその実質的な後継法案が大統領署名済み
- No: 署名されていない（修正中、審議中、廃案含む）
- 判定ソース: congress.gov、ホワイトハウス公式発表、主要法律事務所のトラッカー
- 解釈ポイント: 大幅修正版は同じ法案か？他法案への組み込みは？→ AIが文脈判断

### AI判定エージェント構成
| Agent | モデル | 役割 |
|---|---|---|
| A | Gemini 3 Pro | 総合#1、広い文脈理解 |
| B | Claude Opus 4.5 | 推論/判断トップ |
| C | GPT-5.2 | 深い分析力 |

合議: 3モデルに同じプロンプト → 各自 Yes/No + 確信度(0-100) + 根拠 → 2/3以上一致で確定
